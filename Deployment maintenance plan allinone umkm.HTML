02 · Jaring Pengaman

Rollback Strategy

4 trigger
Pendekatan Deployment

Rolling update dengan health check (didukung native oleh Docker Compose/Coolify) — cukup untuk skala tim & traffic saat ini. Migrasi ke blue-green deployment dipertimbangkan saat traffic bertambah signifikan di Fase 3, bukan di awal, karena kompleksitas tambahan tidak sepadan dengan skala saat ini.

Versioning

Semantic versioning (vMAJOR.MINOR.PATCH) + git tag per rilis production. Docker image di-tag ganda: git-sha (untuk rollback presisi ke commit tertentu) dan versi semver (untuk komunikasi rilis ke tim/pengguna).

Trigger Rollback

Bug Critical baru ditemukan pasca-deploySesuai definisi severity di Testing & QA ChecklistRollback segera — tidak menunggu investigasi akar masalah selesai dulu
Lonjakan komplain penggunaLaporan manual dari CS/PM yang mengindikasikan dampak luasTim putuskan rollback berdasarkan dampak bisnis, bukan hanya data teknis

Perhatian khusus — migrasi database & arsitektur offline-first Karena client menyimpan salinan data lokal yang sinkron ke server (§4 Tech Spec), rollback skema database secara langsung berisiko tinggi begitu client mulai sinkron dengan skema baru. Gunakan pola expand-contract: migrasi menambah kolom/tabel baru tanpa menghapus yang lama dulu → deploy → baru di rilis berikutnya hapus struktur lama setelah dipastikan tidak ada client offline lama yang masih memakainya. Hindari rollback migration langsung di production jika sudah ada data baru yang masuk ke skema baru — lebih aman melakukan hotfix forward.
Feature flag sebagai kill switch Fitur besar/berisiko (mis. prediksi AI, integrasi marketplace baru) diaktifkan lewat flag sederhana (env var atau tabel feature_flags), sehingga bisa dimatikan cepat tanpa rollback deployment penuh saat bermasalah — terutama berguna untuk fitur Fase 3 yang bergantung pada API pihak ketiga.

Runbook per Skenario
01

Server/API Down

Gejala

Alert Uptime Kuma, pengguna tidak bisa transaksi sama sekali

Diagnosis
  • Cek status container di Coolify
  • Cek log Fastify terakhir
  • Cek koneksi ke database
Resolusi
  • Restart container
  • Jika masih gagal → rollback ke image sebelumnya (§02)
  • Jika VPS provider bermasalah → cek status page provider
02

Koneksi Database Bermasalah

Gejala

Error 500 masal, log "connection refused/timeout"

Diagnosis
  • Cek jumlah koneksi aktif vs limit Postgres
  • Cek query yang berjalan lama (slow query)
Resolusi
  • Restart connection pool
  • Hentikan/optimalkan query bermasalah
  • Scale up resource DB bila perlu
03

Payment Gateway/QRIS Gagal

Gejala

Banyak transaksi QRIS macet di status "menunggu konfirmasi"

Diagnosis
  • Cek status page payment gateway
  • Cek log webhook callback masuk
Resolusi
  • Informasikan mode fallback tunai via banner in-app
  • Hubungi support payment gateway
  • Verifikasi ulang transaksi tertunda begitu API pulih
04

Sync Engine Backlog Menumpuk

Gejala

Dashboard sync backlog tinggi, banyak device belum ter-update

Diagnosis
  • Cek Bull Board untuk job sync yang stuck
  • Cari payload corrupt yang macet di antrean
Resolusi
  • Isolasi item gagal (sesuai edge case kategori C, QA Checklist)
  • Scale worker sync
  • Restart worker jika perlu
05

WhatsApp Business API Gagal Kirim

Gejala

Reminder piutang / invoice digital tidak terkirim ke pelanggan

Diagnosis
  • Cek rate limit di dashboard WhatsApp Business Platform
  • Cek status verifikasi nomor bisnis
Resolusi
  • Fallback ke email/unduh PDF manual (sesuai edge case E1)
  • Hubungi dukungan WhatsApp Business jika akun bermasalah
06

Sertifikat SSL Mendekati Kadaluarsa

Gejala

Alert otomatis 30/7 hari sebelum expired

Diagnosis
  • Pastikan auto-renewal Let's Encrypt aktif di Coolify
Resolusi
  • Trigger renewal manual jika auto-renewal gagal
  • Verifikasi sertifikat baru aktif di seluruh domain/subdomain
07

Disk Space Penuh di VPS

Gejala

Alert disk usage >80%

Diagnosis
  • Cek ukuran log lama & Docker image menumpuk
Resolusi
  • Bersihkan log lama & prune Docker image tidak terpakai
  • Pindahkan file backup ke object storage (Cloudflare R2)
